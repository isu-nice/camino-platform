## 1. 동시 예약 시 재고 초과 판매 / 500 에러

### 문제 상황
알베르게(순례자 숙소) 침대 5개에 20명이 동시에 예약을 시도하는 상황을 재현하고 싶었음. 실제 카미노 데 산티아고에서 인기 구간(사리아 등) 알베르게는 선착순 배정이라, 오픈 시간에 트래픽이 몰리는 문제가 실제로 존재함.

### 재현 방법
- 부하테스트 도구 **k6**로 재고 5개인 알베르게에 **20 VU(가상 사용자)가 동시에 POST 요청**
- 예약 로직에는 아직 명시적인 동시성 제어(비관적 락, 재시도 등)를 걸지 않은 상태에서 우선 테스트

```javascript
export const options = {
  vus: 20,
  iterations: 20,
};
```

### Before (낙관적 락만 있고, 충돌 처리는 안 한 상태)

**결과**

| 지표 | 값 |
|---|---|
| 총 요청 | 20건 |
| 성공(200) | 4건 |
| 실패(500) | 16건 |
| 평균 응답시간 | 약 209ms |

**원인 분석**
- `Albergue` Entity에 JPA `@Version` 필드를 두어 낙관적 락은 이미 적용된 상태였음
- 여러 트랜잭션이 동시에 같은 row(재고)를 읽고 수정하려 하면서 `OptimisticLockException` 발생
- **문제는 재고 초과 판매 자체는 막혔지만(다행), 충돌이 사용자에게 그냥 "500 Internal Server Error"로만 노출됐다는 점**
  - 낙관적 락이 "동시 수정 충돌 감지"는 해주지만, "감지된 충돌을 어떻게 처리할지"는 별도로 설계해야 한다는 걸 실측으로 확인

### 해결

**1차: 409 Conflict로 명확히 응답**
- `@RestControllerAdvice`로 전역 예외 처리기(`GlobalExceptionHandler`)를 두어 `OptimisticLockingFailureException`을 캐치
- 500 대신 409 Conflict + `"다른 요청과 예약이 충돌했습니다"` 메시지로 응답하도록 변경

**2차: 재시도 로직 추가**
- 409로 명확히 응답하게는 됐지만, 여전히 재고가 남아있는데도(5개 중 2개만 성공) 타이밍 문제로 실패하는 사용자가 많다는 걸 발견
- 트랜잭션 경계를 분리(`reserve()` 재시도 루프 / `doReserve()` 실제 트랜잭션)해서, `OptimisticLockingFailureException` 발생 시 최대 5회까지 짧은 대기 후 재시도하도록 구현
- 재시도 간격은 `10ms * 시도횟수`로 점증시켜, 재시도들이 다시 한번에 몰려 충돌하는 걸 완화

### After

| 단계 | 성공 | 실패 | 실패 원인 |
|---|---|---|---|
| Before (락만, 재시도 없음) | 4 / 20 | 16 / 20 (500) | 타이밍 충돌로 억울하게 실패 |
| 1차 (409 응답만) | 4 / 20 | 16 / 20 (409) | 응답은 명확해졌지만 성공률은 그대로 |
| 2차 (재시도 추가) | **5 / 20** | 15 / 20 (409) | **재고(5개)만큼 정확히 성공, 나머지는 진짜 재고 소진** |

재시도 로직 적용 후, 성공 건수가 실제 재고 수(5개)와 정확히 일치하는 것을 확인함 — 더 이상 "타이밍 문제로 인한 억울한 실패"가 없고, 실패는 전부 "진짜 자리가 없어서" 발생하는 정상적인 케이스로 좁혀짐.

### 락 전략 비교: 낙관적 락+재시도 vs Redis 분산 락

재시도 로직으로 정합성 문제는 해결했지만, 동시 요청 수가 훨씬 커지면(수백~수천) 재시도가 재시도를 부르며 DB 부하가 커질 수 있다는 우려가 있어 Redis 분산 락(Redisson)과 비교 실험을 진행함.

**구현**
- Redisson의 `RLock`으로 알베르게 단위 락(`lock:albergue:{id}`) 적용
- `tryLock(waitTime=2s, leaseTime=5s)`: 최대 2초 대기 후 락 획득 시도, 획득 시 최대 5초간 보유 (보유 프로세스가 비정상 종료되어도 자동 해제되도록 leaseTime 설정)
- 락 안에서는 재시도 없이 단순하게 재고 감소 + 예약 생성만 수행

**비교 결과**

| 방식 | 동시 요청(VU) | 성공 | 실패 | 평균 응답시간 | 정합성 |
|---|---|---|---|---|---|
| 낙관적 락 (재시도 없음) | 20 | 4/20 | 16/20 (500) | 209ms | ❌ 타이밍에 따라 부정확 |
| 낙관적 락 + 재시도 | 20 | 5/20 | 15/20 (409) | 253ms | ✅ 재고와 정확히 일치 |
| Redis 분산 락 | 50 | 5/50 | 45/50 (409) | 393ms | ✅ 재고와 정확히 일치 (더 높은 부하에서도) |

**트레이드오프 분석**
- 두 방식 모두 최종 정합성(재고 수만큼만 성공)은 달성했지만 성격이 다름
- 분산 락은 순차 처리를 강제해서 개별 응답시간은 더 길지만(락 대기 시간 포함), 경합이 심해질수록(동시 요청이 훨씬 많아질수록) DB에 불필요한 쓰기 재시도가 쌓이지 않아 더 안정적일 것으로 예상
- 반대로 충돌이 드문 상황(트래픽이 적당히 분산된 경우)에서는 락 대기 오버헤드가 없는 낙관적 락+재시도가 더 가벼움
- **결론**: 알베르게 오픈런처럼 특정 순간에 경합이 극심하게 몰리는 시나리오에는 Redis 분산 락이 더 적합하다고 판단, 최종적으로 분산 락 방식을 채택

### 배운 점
- JPA `@Version`은 동시성 "충돌 감지" 도구일 뿐, "충돌 대응"까지 자동으로 해주지 않는다는 걸 직접 겪으며 이해함
- 문제를 추측이 아니라 k6로 수치화해서 재현했기 때문에, 이후 해결책 적용 후에도 같은 방식으로 Before/After를 객관적으로 비교할 수 있는 기반을 마련함
- 낙관적 락은 "충돌 시 즉시 실패"가 기본 동작이라, 트래픽이 몰릴수록 "진짜 재고 부족"과 "타이밍 문제로 인한 실패"가 섞여서 사용자 경험을 해칠 수 있다는 걸 확인함. 짧은 재시도를 더하는 것만으로도 이 둘을 효과적으로 구분할 수 있었음
- 낙관적 락+재시도와 분산 락 모두 정합성은 달성할 수 있지만, 응답시간·부하 안정성 트레이드오프가 다르다는 걸 실측으로 확인함 — "정답은 하나가 아니라 트래픽 패턴에 따라 선택이 달라진다"는 걸 체감함
- 분산 락 도입 시 leaseTime을 반드시 설정해야, 락을 잡은 프로세스가 비정상 종료돼도 다른 요청들이 영원히 막히지 않는다는 걸 (실제로 겪기 전에) 미리 알고 대비함

## 2. DB 커넥션 풀 고갈

### 문제 상황
락 로직이 있는 예약 API는 이미 순차 처리되고 있어 커넥션 풀 문제가 잘 드러나지 않았음. 대신 단순 조회 API(`GET /api/albergues`)에 지속적인 부하를 걸어, 풀 크기가 부족할 때 어떤 일이 생기는지 별도로 재현함.

### 재현 방법
- HikariCP 풀 크기를 일부러 2로 축소 (`maximum-pool-size: 2`), 타임아웃도 250ms로 단축
- k6로 VU 20, 10초간 지속 부하 (`GET /api/albergues`)

**시행착오**: 처음엔 로컬 Postgres가 같은 머신에서 돌아 쿼리가 1ms 이내로 끝나버려서, 풀을 2로 줄여도 대기가 타임아웃까지 이어지지 않아 실패가 재현되지 않았음. 실제 운영 환경(DB가 네트워크 너머에 있어 쿼리에 수십 ms가 기본으로 걸리는 상황)을 흉내내기 위해, `@Transactional` 메서드 안에 `Thread.sleep(100)`을 넣어 커넥션을 인위적으로 오래 점유하도록 만든 뒤에야 정상적으로 재현됨.

### Before (풀 크기 2, 타임아웃 250ms)

| 지표 | 값 |
|---|---|
| 성공률 | 25.75% (188/730) |
| 실패율 | 74.24% (542/730, connection timeout) |
| 평균 응답시간 | 276.63ms |
| p95 응답시간 | 349.31ms |
| 처리량 | 71.3 req/s |

서버 로그에 `Connection is not available, request timed out after 250ms` 반복 발생.

### 해결
- 풀 크기를 기본 권장값인 10으로 복구
- 무조건 크게 잡지 않고 기본값에서 시작한 이유: DB 자체의 `max_connections` 제한과 커넥션당 메모리 비용을 고려하면, 무작정 키우기보다 트래픽 패턴에 맞춰 실측 후 조정하는 게 맞다고 판단

### After (풀 크기 10, 타임아웃 3000ms, 동일하게 100ms 인위적 지연 유지)

| 지표 | Before (풀 2) | After (풀 10) |
|---|---|---|
| 성공률 | 25.75% | **100%** |
| 평균 응답시간 | 276.63ms | 218.7ms |
| p95 응답시간 | 349.31ms | 320.22ms |
| 처리량 | 71.3 req/s | 90.8 req/s |

### 배운 점
- 커넥션 풀 부족은 항상 에러로 나타나는 게 아니라, 먼저 "응답 지연"으로 나타나고 타임아웃을 넘겨야 비로소 에러가 된다는 걸 확인함. 그래서 실무에서 "느려졌는데 에러 로그는 없다"는 애매한 상황의 원인이 될 수 있음을 체감
- 로컬 환경이 운영 환경보다 지나치게 빠르면(같은 머신 내 DB), 오히려 병목을 재현하기 어려울 수 있다는 걸 겪음. 실제 네트워크 지연을 흉내내는 인위적 지연을 넣고 나서야 문제를 제대로 재현할 수 있었음
- 풀 크기는 클수록 무조건 좋은 게 아니라, DB의 최대 커넥션 제한과 트레이드오프가 있다는 걸 인지하고 접근함

## 3. N+1 쿼리 문제

### 문제 상황
예약 목록 조회 API(`GET /api/reservations`)에서 각 예약이 참조하는 알베르게 이름까지 응답에 포함시켜야 했음. `Reservation`의 `@ManyToOne albergue` 필드가 지연 로딩(LAZY)이라, 목록을 순회하며 `albergue.getName()`을 호출할 때마다 개별 쿼리가 추가로 발생하는 구조였음.

### 재현 방법
- 서로 다른 알베르게 8곳에 걸쳐 예약 10건을 생성
- `show-sql: true`로 실제 나가는 쿼리를 로그에서 직접 확인

**시행착오**: 처음엔 테스트 데이터가 전부 같은 albergue_id를 참조해서, JPA 영속성 컨텍스트(1차 캐시)가 이미 조회한 id는 재사용해버려 N+1이 로그상 드러나지 않았음. 서로 다른 albergue를 참조하는 예약으로 데이터를 재구성한 뒤에야 문제가 명확히 나타남.

### Before

예약 10건 조회 시 발생한 쿼리:
```
select ... from reservations r1_0                              -- 1회
select ... from albergues a1_0 where a1_0.id=?                 -- 8회 (서로 다른 albergue 개수만큼)
```
**총 쿼리 수: 9회** (1 + N, N = 8)

### 해결
`ReservationRepository`에 `JOIN FETCH`를 사용한 커스텀 쿼리를 추가해서, 연관된 `Albergue`를 지연 로딩 대신 즉시 하나의 쿼리로 함께 가져오도록 변경.

```java
@Query("SELECT r FROM Reservation r JOIN FETCH r.albergue")
List<Reservation> findAllWithAlbergue();
```

### After

동일한 데이터(예약 10건, 8개 알베르게)로 재조회:
```
select r1_0..., a1_0... from reservations r1_0 join albergues a1_0 on a1_0.id=r1_0.albergue_id
```
**총 쿼리 수: 1회**

| 지표 | Before | After |
|---|---|---|
| 총 쿼리 수 | 9회 | **1회** |

### 배운 점
- 지연 로딩(LAZY) 자체는 불필요한 데이터를 안 가져오기 위한 좋은 기본값이지만, 목록 조회처럼 연관 데이터를 전부 써야 하는 상황에서는 오히려 쿼리 수를 폭증시킬 수 있다는 걸 확인함
- JPA의 영속성 컨텍스트(1차 캐시)가 동일 트랜잭션 내 중복 조회를 막아주기 때문에, 테스트 데이터가 우연히 겹치면 N+1 문제가 가려질 수 있다는 것도 실험 중에 겪음 — 문제를 재현할 때 데이터 다양성이 실제로 중요하다는 걸 체감
- `JOIN FETCH`는 단일 연관관계(`@ManyToOne`)에는 안전하지만, 컬렉션(`@OneToMany`) fetch join과 페이징을 같이 쓰면 메모리 페이징 문제가 생길 수 있어 상황에 따라 다르게 접근해야 함을 인지

## 4. 캐시 스탬피드

### 문제 상황
알베르게 상세 조회(`GET /api/albergues/{id}`)에 Redis 캐시(TTL 3초)를 적용했으나, TTL이 만료되는 순간 동시에 몰린 요청들이 전부 캐시 미스로 판단해 동시에 DB를 조회하는 문제가 있을 것으로 예상하고 재현함.

### 재현 방법
- `@Cacheable`로 Redis 캐시 적용, TTL 3초로 설정 (재현 속도를 위해 실제 운영값보다 짧게 설정)
- k6로 VU 30, 15초간 지속적으로 동일 리소스(`/api/albergues/1`) 조회
- `show-sql: true`로 서버 로그에서 실제 DB 쿼리 발생 시점 관찰

**시행착오**:
- Redis에 Entity를 캐싱하는 과정에서 두 가지 직렬화 문제를 순서대로 겪음
  1. 기본 Java 직렬화(`Serializable`) 미구현으로 저장 자체가 실패 → JSON 직렬화(`GenericJackson2JsonRedisSerializer`)로 전환
  2. JSON에 타입 정보가 없어 역직렬화 시 `Albergue`가 아닌 `LinkedHashMap`으로 복원되는 `ClassCastException` 발생 → `ObjectMapper`에 `activateDefaultTyping` 적용해 타입 메타데이터를 함께 저장하도록 해결

### Before

TTL 3초 동안 이론상 조회는 1회면 충분하지만, 실제로는 TTL 만료 순간마다 동시에 몰린 요청 수만큼(로그상 한 타이밍에 20회 이상) `Hibernate: select ... from albergues` 쿼리가 연속으로 발생함을 확인.

### 해결
이미 예약 동시성 문제 해결에 사용했던 Redis 분산 락(Redisson)을 캐시 재구축 보호에도 재사용:
- 캐시 미스 발생 시, 락을 획득한 요청 한 건만 DB를 조회하고 캐시를 채움
- 락을 획득하지 못한 나머지 요청은 짧게 대기 후 캐시를 재확인해서 결과를 공유
- 더블 체크(double-checked locking) 패턴 적용: 락 획득 직후에도 캐시를 한 번 더 확인해, 대기 중 이미 다른 요청이 채운 경우 중복 조회를 방지

### After
동일한 부하(VU 30, 15초) 재실행 결과, TTL 만료 시점마다 DB 쿼리가 정확히 1회씩만 발생 (약 3초 간격으로 조용히 한 번씩만 로그에 찍힘). 요청 동시성과 무관하게 DB 부하가 상수로 유지됨을 확인.

### 배운 점
- 캐시를 도입하는 것 자체보다 "캐시가 비는 순간을 어떻게 보호할 것인가"가 실제로 더 중요한 설계 지점이라는 걸 체감함
- 한 번 구축한 인프라(Redis 분산 락)를 서로 다른 문제(예약 동시성, 캐시 스탬피드)에 재사용할 수 있었음 — 특정 기술을 하나의 용도로만 좁게 이해하지 않고, "동시 접근 제어"라는 더 일반적인 문제 해결 도구로 이해하게 됨
- Redis에 객체를 캐싱할 때 직렬화 방식(Java 기본 직렬화 vs JSON)과 타입 정보 보존 여부를 명시적으로 신경 써야 한다는 걸 실제 에러를 겪으며 이해함

## 5. k8s 리소스 미설정으로 인한 OOMKilled

### 문제 상황
지금까지 albergue-service를 docker-compose로만 실행해왔고, 로컬 kind 클러스터에 실제로 배포한 적이 없었음. k8s Deployment에 리소스 request/limit을 지정하지 않거나 너무 낮게 잡으면 어떤 문제가 생기는지 직접 재현함.

### 재현 방법
- `k8s/albergue-deployment.yaml`에 memory limit을 128Mi로 의도적으로 낮게 설정 (Spring Boot JVM이 정상 기동하기에 부족한 값)
- Postgres/Redis도 k8s Deployment/Service로 함께 배포해서, docker-compose 없이 순수 k8s 환경에서 재현

### Before

```
State:          Terminated
  Reason:       OOMKilled
  Exit Code:    137
Restart Count:  4
```

2분 44초 사이 4번 재시작을 반복하다 `BackOff` 상태(`CrashLoopBackOff`)에 빠짐. Exit Code 137은 `128 + SIGKILL(9)`로, 메모리 초과로 강제 종료됐음을 나타냄.

### 해결
리소스 request/limit을 Spring Boot 애플리케이션이 안정적으로 기동 가능한 수준으로 재설정:

```yaml
resources:
  requests:
    memory: "256Mi"
    cpu: "250m"
  limits:
    memory: "512Mi"
    cpu: "500m"
```

- `requests`: 스케줄링 시 최소 확보량 (k8s가 Pod를 배치할 노드를 고를 때 사용)
- `limits`: 초과 시 강제 종료(메모리) 또는 스로틀링(CPU)되는 상한선

### After

```
State:          Running
Ready:          True
Restart Count:  0
```

재배포 후 즉시 안정적으로 기동, 재시작 없이 지속적으로 Running 상태 유지.

### 배운 점
- 리소스 limit 미설정/과소 설정은 로컬 docker-compose 환경에서는 절대 드러나지 않는 문제라는 걸 확인함 — docker-compose는 호스트 메모리를 사실상 제한 없이 쓰지만, k8s는 명시된 한도를 엄격하게 강제하기 때문
- 그래서 "로컬에서 됐으니 배포도 되겠지"라고 가정하면 안 되고, 실제 배포 환경(k8s)에서 별도로 검증하는 과정이 반드시 필요하다는 걸 체감함
- requests와 limits의 역할이 다르다는 걸 실습으로 이해함 — requests는 스케줄링 기준, limits는 실제 강제 상한
